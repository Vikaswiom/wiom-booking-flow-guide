Critical Insight

This is the biggest single leak in the entire acquisition funnel. Of first-time users who clicked "Pay ₹100", only 18.4% (PRE) / 24.9% (POST) actually completed payment. These users reached the payment page but dropped off without completing — not necessarily payment failures. Likely causes: last-second hesitation, price shock, back-button, app close, or abandonment before initiating UPI/card entry.

📘 How To Read The Matrix
  • 🔮 Delay / Churn Criteria — the delay message threshold from the booking flow (e.g., "48h without partner interest"). Customers hitting this threshold are predicted likely to churn.
  • 📊 Stage Rates (% of stage entries) — shows what fraction of customers who reached each stage ultimately got cancelled. All numbers are percentages of stage entries, so you can directly compare stages.
  • By Customer = self-cancelled by the user. By System = `cops_unserviceable` / cops-initiated auto-cancellation. Actual Total = sum of both.
  • Predicted vs Actual: if Predicted ≈ Actual → the delay criteria is a reliable churn signal. If Actual > Predicted → customers cancel BEFORE the threshold (suggest tighter criteria). If Predicted > Actual → many stuck customers don't cancel (threshold is loose or they're waiting).

📍 Change 1 · Location Clarity Before PaymentMulti-signal serviceability check + presence confirmation → no more post-payment rejections
📍
The Problem
Post-payment serviceability rejections erode customer trust
Current flow checks serviceability only AFTER ₹100 payment · False positives + false negatives
Why now

Wiom's current booking flow uses a single GPS reading at app launch, matched against a broad area-level coverage map. The precise pin-level check happens only AFTER the customer has paid. When these disagree → booking cancelled, money refunded, but trust damage is already done. Additionally, GPS drift or customers not being at the install location → wrongly rejected serviceable customers (false negatives).

✓ What's new
  • Multi-signal serviceability — GPS + backend Wiom net proximity detection (active connections within ~10m / ~20m / ~50m)
  • Pre-payment recheck — background GPS capture when customer taps "₹100 dekar book karein"
  • Presence confirmation screen — "Kya aap abhi wahi hain jaha net lagwana hai?" before payment proceeds
  • Reminder flow — "Jab ghar pahunchein bata dijiye" → push notification at 9 PM → deep-link back to confirmation
  • Trust recovery — auto-refund for rare post-payment failures + "notify me" CTA
